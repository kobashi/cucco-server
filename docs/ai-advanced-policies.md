# 上位AI 実装案集(教材)

Mock AIの3基本方策(必ずチェンジ / 必ずノーチェンジ / 人数×ランクのマトリクス。
`clients/mock_ai/policies.py`)より強いAIを作るための実装案集。**サーバー側では
実装しない**。ゼミ生が自分のAIクライアントを設計する際の出発点・ヒント集として
使うこと。プロトコルの詳細は`docs/ai-client-guide.md`、ルールは
`docs/rules/final_rules.md`を参照。

どの案も、AIクライアントが受信できる**公開情報だけ**で成立する。カンニング
(他人の手札の直接参照)は物理的に不可能な設計になっている。

## 0. 上位AIが考慮すべき6つの観点

| # | 観点 | 一言で |
|---|---|---|
| 1 | 自分が最弱である確率 | 「今の手札、実は負けてる?」を数値で持つ |
| 2 | チェンジ後の改善確率 | 交換相手から何が来るかの分布を持つ |
| 3 | 山札交換の入手カード制限 | 山札は対人交換と分布が全く違う(下記) |
| 4 | 宣言経緯からの相手手札推定 | チェンジ→ノーチェンジの並びは強い証拠 |
| 5 | 最弱札の伝播と消滅 | 弱札は山札方向へ流れるか失格と共に消える |
| 6 | クク宣言タイミングの損得 | 即時宣言が常に最善とは限らない |

## 案A: カウンティング確率AI(難易度: 低)

**ねらい**: 固定閾値(matrix)を「その場で計算した確率」に置き換える。
上位AIの共通土台になる。

**必要な情報と管理方法**:
- 未知カード集合: 各ランク2枚×22種の多重集合から、自分の手札と観測済みの
  捨て札(`deal_result`の`discarded_cards`、即時公開設定なら
  `player_disqualified`の`card`も)を引いたもの
- `deck_reshuffled`が来たら捨て札が山札に戻る=カウンティングをリセットする
  (`deck_remaining_count`も再構築後の値になる)

**アルゴリズムの骨子**:
1. 手番が来たら、未知カード集合のうち自分の手札より弱いカードの割合 p を計算
2. 生き残り人数 n に対し、「全員が自分より強い」確率 ≈ (1−p)^(n−1) を推定
   (厳密には非復元抽出。まずは独立近似で十分、精密化は発展課題)
3. 「自分が最弱の確率」がしきい値を超えたらチェンジ

**実装のヒント**:
- `MockAI`クラス(`clients/mock_ai/mock_ai.py`)のイベントハンドラに
  カウンティング用の辞書 `{rank: 残枚数}` を足すだけで始められる
- 観点3(山札交換)の分布は別枠で作る:
  - 入手不可: クク(拒否・手札維持)、馬・家(引き直し)、人間(引いた本人が
    失格)、猫(自分の手札の来歴元が失格。**自分の配布札を持ったままなら自滅**)
  - 道化は例外的に**最強**として入手(大当たり)
  - 通常入手は数字札・桶・仮面・獅子のみ。分布は山札構成から直接計算できる
  - つまり「山札交換の期待値」は、山札に道化が残っているか・人間が残っている
    かで大きく振れる。ここがカウンティングの威力の見せ場

**検証**: 評価モードでmatrix方策と100ゲーム以上対戦させ、勝率・平均順位で比較。

## 案B: ベイズ推論AI(難易度: 中)

**ねらい**: 案Aの「未知カードは一様」という仮定を捨て、相手の行動から
相手ごとの手札分布(事後分布)を推定する。6観点すべてを明示的に扱える本命。

**データ構造**: プレイヤーごとに長さ22のベクトル(各ランクの保持確率)。
ディール開始時は全員一様、イベントごとに更新して正規化する。

**更新ルールの例(尤度の設計が本体)**:
- `no_change_declared`: そのプレイヤーが弱札を持つ確率を下げる。
  「matrix方策ならこのランクでノーチェンジする確率」を尤度に使うのが簡単
- `exchange_result`(成立): **両者の事後分布を入れ替える**。カード値は非公開
  でも「交換が起きた」事実は公開なので、推定ごと交換すればよい(観点5の
  「弱札ウェーブ」追跡はこれで自動的に実現する)
- `exchange_result`(拒否): 人間・猫は確定情報(そのランクに確率1)。馬/家は
  `horse_house_reveal`設定次第で確定または2択
- **観点4の並び**: プレイヤーiがチェンジ→受け手i+1がノーチェンジ、の並びを
  見たら「iが差し出した弱札より、iが受け取った札の方がさらに弱い」方向に
  iの分布を更新する(受け手が受領札をそのまま保持した=差し出した札の方が
  受領札より強かった、という推論)
- `cucco_declared`: 宣言者はクク確定(ただしディールは即終了)
- 来歴(`provenance_map`)は公開情報。猫の失格予測にそのまま使える

**判断**: 各相手の分布から「自分が最弱の確率」を計算(案Aの式の p を相手ごと
の分布で置き換える)。チェンジ判断・クク宣言判断(観点6: 自分より前の手番に
未行動プレイヤーが残っているか+今凍結した場合の敗者分布)に使う。

**実装のヒント**:
- 更新式は「事前分布 × 尤度 → 正規化」の繰り返しだけ。numpyすら不要
- 尤度モデルは最初は雑でよい(matrix方策を全員に仮定)。相手の実際の行動履歴
  から尤度を較正するのは発展課題
- 落とし穴: 失格者の手札の扱い(遅延公開設定では消えたカードが未知のまま)、
  reshuffle後のリセット、自分がターゲットにされた交換での自分の分布更新漏れ

## 案C: モンテカルロ・シミュレーションAI(難易度: 中〜高)

**ねらい**: 確率式を立てる代わりに「あり得る世界を大量に作って数える」。
連鎖・山札交換・クク宣言の損得(観点3・5・6)をルールごと再現できる。

**アルゴリズムの骨子**:
1. 観測と矛盾しない手札配置をサンプリング(案Bの事後分布を重みに使うと精度向上)
2. 各候補行動(チェンジ / ノーチェンジ / クク宣言可能なら宣言・パス)について、
   相手全員がmatrix方策と仮定してディール末尾まで進行をシミュレート
3. 行動別の「自分が敗者にならない確率」(またはチップ期待値)を集計し最良を選択

**実装のヒント**:
- シミュレータは自作しなくてよい。`cucco.domain.deal.Deal`は純粋関数的な
  同期エンジンなので、`pip install -e .`済みならクライアント側からそのまま
  `Deal(participants, dealer_id, Deck.from_fixed_order(...), config)`で
  仮想ディールを回せる(テストコード`tests/unit/domain/`が使用例の宝庫)
- AI用手番タイムアウトは10秒。この規模なら1000プレイアウトは余裕で収まるが、
  クク窓は2秒なので宣言判断だけは軽くしておくこと
- 乱数シードを固定すればデバッグ再現が容易(サーバー側の行動ログと同じ思想)

## 案D: チップ期待値AI(難易度: 低、A〜Cへの拡張層)

**ねらい**: 「このディールに勝つ」ではなく「ゲーム終了時のチップ」を最大化する。
判断基準の層なので、A〜Cのどれにでも後付けできる。

**考えどころ**:
- 子供の時間(1〜3ディール目)の敗北は`deal_number`枚の支払いで済む。序盤は
  安く負けてよく、無理なチェンジでリスクを取る必要がない
- 大人の時間(4ディール目以降)の敗北は即脱落。リスク許容度を切り替える
- `continue_prompt`への応答も期待値問題: 支払額 vs 「残留した場合にポットを
  取れる確率×ポットサイズ」。残りチップが支払額ギリギリなら辞退も合理的
- 終了条件(`chips_zero` / `round_limit`)で最適戦略が変わる。chips_zeroでは
  「誰かが0になった時点で順位確定」なので、自分が2位以上ならゲームを終わらせる
  方向に打つ、という順位保全の判断もあり得る

## 案E: 強化学習AI(難易度: 高、発展課題)

**ねらい**: 方策を人手で設計せず、自己対戦から学習する。ゼミの研究テーマ向き。

**この基盤が学習環境としてそのまま使える理由**:
- 評価モード(`mode: "evaluation"`)で数百ゲームの自動連続対局が回る
- 行動ログ(JSON Lines+シャッフルシード)で全対局が決定論的に再現できる
  → オフライン学習・リプレイバッファに使える
- `evaluation_summary`の勝率・平均順位がそのまま評価指標になる

**最初の一歩の例**: 状態を(自分のランク, 生き残り人数, 手番位置, 子供/大人)
程度に粗く離散化したQ学習から。報酬はディール単位のチップ増減。相手は
matrix方策で固定してから自己対戦に進む(いきなり自己対戦は非定常で難しい)。

## 共通: 検証の作法

1. 評価モードで新AIをmatrix方策×3と対戦させる(座席バイアスは自動で除去される)
2. `game_count`は100以上。20ゲーム程度では勝率の差が偶然と区別できない
3. `evaluation_summary`の`disqualification_rate`は診断に有用: 高すぎるなら
   人間・猫への突撃(無謀なチェンジ)を疑う
4. 行動ログを読み、「そのチェンジ判断は事後分布から見て妥当だったか」を
   1ディール単位でレビューする。これが一番の教材になる
