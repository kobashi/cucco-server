# Cucco ネット対戦サーバー 要求仕様書

## 1. 概要

### 1.1 目的

カードゲーム「Cucco」(グランペール版、「カンビオ」ルール)のネット対戦サーバーを構築する。ゲーム研究ゼミ内での運用を目的とし、プレイヤーは人間・AIの両方に対応する。

### 1.2 想定利用者

- ゼミ生(人間プレイヤーとして、またAIプレイヤーの開発者として参加)
- AIプレイヤー(ゼミ生がCodex/Gemini等を利用してPythonで実装)
- 段階的に開放される学外参加者(卒業生等)

### 1.3 運用フェーズ

1. 開発・初期運用: iMac上でローカル起動し、学内ネットワークからアクセス
2. 安定運用後: Cloudflare Tunnel等を用いて限定的に学外へ公開し、卒業生等の参加を許可

### 1.4 用語集

| 用語 | 意味 |
|---|---|
| ディール | カードを配ってから「オープン」で決着するまでの一勝負 |
| ポット | 最後の1人が残るまでディールを繰り返す単位。勝者がチップを総取りする |
| ゲーム | ポットを繰り返し、終了条件(所持チップ0枚 or 既定ラウンド数)に達するまでの一連の対局 |
| 子供の時間 | ポット内の1〜3ディール目。敗者はディール数に応じたチップを支払って参加を継続する |
| 大人の時間 | ポット内の4ディール目以降。敗者はそのポットから脱落する |
| カンビオ/ノンカンビオ | 手番プレイヤーが右隣のプレイヤーとの交換を要求する/しない宣言 |
| クク宣言 | クク札の所持者が、いつでもディールを即終了させられる宣言 |
| 卓(テーブル) | 1つの独立したゲームが進行する単位。複数の卓が並行して稼働できる |
| プレイルームID | 卓に参加するための招待コード |

## 2. 機能要件

### 2.1 ゲームルール

Cucco(カンビオルール、グランペール版)の完全なルールは [`docs/rules/final_rules.md`](rules/final_rules.md) に定義する。要点は以下の通り。

- カード構成: 22種類×2枚=44枚(数字札0〜12、特殊札: 家・猫・馬・人間・クク、弱札: 桶・仮面・獅子・道化)
- 1ディールの流れ: 配布→親の「どうぞ」宣言→手番順にカンビオ/ノンカンビオ→親は山札と交換→オープン→最弱者が敗者
- 特殊札(道化・人間・馬・家・猫・クク)固有の効果、および山札交換時の連鎖処理
- チップ管理: 子供の時間/大人の時間によるポットの進行、自動復活、最終勝者なし(全滅)時の持ち越し
- ゲーム開始時に選択可能な設定項目(終了条件、失格者カードの開示タイミング、馬/家の開示方法)

ルールの参考資料・バリエーション・出典は同ファイル内に記録済み。カードのイラスト・アイコンは実装において一切参照しない(名称・数値・構造のみ使用)。

### 2.2 プレイヤー種別

| 種別 | 説明 |
|---|---|
| 人間 | 別途用意するUIを通じて手動で操作する |
| AI | ゼミ生が実装するPythonクライアント。通信プロトコルに従って自動で対局する |
| 観戦者 | ゲームに参加せず、公開情報のみを閲覧する(手札は見えない) |

### 2.3 通信プロトコル

技術詳細は [`docs/protocol/design.md`](protocol/design.md) に定義する。要点は以下の通り。

- 通信方式: WebSocket、メッセージ形式はJSON、共通エンベロープに`protocol_version`を含む
- クライアント→サーバーのアクション一覧(`identify`, `create_table`, `join_table`, `ready`, `dealer_ready`, `cambio_declare`, `no_change_declare`, `cucco_declare`, `cucco_pass`, `continue_declare`)
- サーバー→クライアントのイベント一覧(状態同期、手番通知、交換結果、失格、ディール/ポット結果、ゲーム終了、山札再構築等)
- 公開ゲーム状態(Public Game State)の完全なデータモデル: 座席・チップ(絶対値)・親・手番・残り山札枚数・捨て札(来歴つき)・宣言履歴・本人の手札(本人にのみ)
- 情報可視性のルール(何が誰に公開されるか)
- クク宣言の機会(`cucco_window`)の仕組みと優先順位
- タイムアウト・切断・再接続・不正操作の扱い(`docs/protocol/decisions.md`に運用方針の詳細)

### 2.4 卓(テーブル)管理

- 卓の作成: プレイヤーが`create_table`で卓を作成し、サーバーが「プレイルームID」を発行する
- 参加方式: 作成者がプレイルームIDを参加者に共有する招待制(オープンなマッチメイキングは行わない)
- 複数卓の並行進行: サーバーは複数の卓を同時にホストし、卓ごとに独立したゲームエンジンで進行する
- 卓の人数上限: プレイヤー最大15人、観戦者最大30人程度(初期値、運用しながら調整)

### 2.5 AI専用高速評価モード

- AIプレイヤーのみで構成される卓モード(`create_table`の`mode: "evaluation"`)
- 人間向けの待機・演出を省き、AIの応答があり次第即座に進行する
- 指定回数のゲームを自動連続実行し(`game_count`)、座席・親をゲームごとにローテーションして座席バイアスを排除する
- 完了時に勝率・平均順位・平均チップ数・失格率などの集計結果を配信する(`evaluation_summary`)

### 2.6 成績記録・永続化

- ゲーム終了ごとに、参加者の成績(卓ID・モード・日時・参加者・最終順位・最終チップ数)をSQLiteに永続化する
- リプレイ・ルール上の疑義の検証・AIの戦略分析のため、山札のシャッフルシードとイベントの時系列ログ(JSON Lines)を別途保存する

## 3. 非機能要件

### 3.1 技術スタック

- 言語: Python
- 通信: WebSocket / JSON
- データ永続化: SQLite(進行中の状態はインメモリ)
- リポジトリ管理: Git、成果物はGitHubで公開

### 3.2 実行環境

- 開発・初期運用: iMac、ローカル起動、学内ネットワークからアクセス
- 安定運用後: Cloudflare Tunnel等で限定公開、学外(卒業生等)からのアクセスを許可

### 3.3 想定規模

- 同時に数卓、卓あたり通常10人程度を想定(プレイ人数の目安は2〜15人、上限は15人)
- 手番タイムアウト: 人間30秒 / AI10秒(サーバー設定値、初期パラメータ)

### 3.4 セキュリティ方針

- プレイルームへの不正参加対策・認証等のセキュリティは、本プロジェクトの規模・用途では考慮しない
- プロトコルの精度は、AI実装側のバグ(不正な操作の送信等)への対処で十分とする

## 4. 開発・検証方針

- 通信対戦の動作確認用のStub、および挙動確認用のMock AIを用意し、プロトコルの実装を検証する
- ゼミ生は本プロトコルに従い、Codex/Geminiなどを利用してPython製のAIプレイヤーを各自実装する
