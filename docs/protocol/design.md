# Cucco 通信プロトコル設計

`docs/rules/final_rules.md`のルールと`docs/protocol/decisions.md`の運用方針に基づく、通信プロトコルの設計。Fable5による7回のレビューを経て、実装可能な水準まで確定している(要求仕様としての全体像は`docs/requirements.md`を参照)。

## 技術スタック

- 言語: Python
- 通信: WebSocket、メッセージ形式はJSON
- サーバー: 開発時はiMacでローカル運用(学内ネットからアクセス)。安定後にCloudflare Tunnel等で限定公開し、卒業生等の学外アクセスを許可
- 想定規模: 小規模(同時に数卓、卓あたり〜10人程度)。複雑な分散構成は不要、単一プロセスで十分。ゲーム進行中の状態はインメモリで管理し、終了時の成績のみSQLiteに永続化する
- AIクライアント: 学生がCodex/Gemini等を使ってPythonで実装。人間クライアントは別途UIを用意(初期はStub/CLIで動作確認)

## アーキテクチャ概要

- サーバーはゲーム状態を一元管理する権威(authoritative)サーバー。クライアントはアクションを要求するのみで、状態そのものはサーバーが確定させ、結果をイベントとして配信する
- **着席順はゲーム開始時にランダム化**される(参加順のままでは座席バイアスが固定されるため)。各ポットの最初の親もランダム(`docs/rules/final_rules.md`「親の交代」)。シャッフルには記録済みシードと同じrngを使うため、リプレイの決定性は保たれる。`state_snapshot`の`seats`はこの着席順で並ぶ
- **卓(Table/Room)**: 1卓 = 1ゲーム(複数ポットの連なり)。サーバーは複数卓を同時にホストし、並行してゲームを進行できる(卓ごとに独立したゲームエンジンインスタンス)
- **プレイルームID**: 卓を作成したプレイヤーに発行されるID。作成者がこのIDを他の参加者に共有し、参加者はIDを指定して`identify`することで卓に加わる(オープンなマッチメイキングは行わない、招待制)
- **ロビー**: 卓に参加する前の待機状態。卓の作成・参加を扱う
- **ゲームエンジン**: ルール(`final_rules.md`)の状態機械を実装するコアロジック。卓ごとに1インスタンス
- **プレイヤーセッション**: 1 WebSocket接続 = 1プレイヤー(人間・AI・観戦者のいずれか)。再接続時は同一プレイヤーIDに紐付けて復帰させる
- **観戦者**: ゲームには参加せず、卓の状態を閲覧できるロール。アクションは送信できない(公開情報のみ受信)
- **評価モード卓**: AIプレイヤーのみで構成し、人間向けの待機・演出を省いて高速にゲームを連続実行する卓。性能評価・ベンチマーク用途
- **リザルトストア**: ゲーム終了時の成績を永続化する記録層(詳細は「永続化・成績記録」を参照)

## 接続・セッションライフサイクル

1. クライアントがWebSocketで接続する
2. クライアントが`identify`アクションを送信する(名前、プレイヤー種別[human/ai/spectator])。サーバーはこの接続に対してセッショントークンを発行し、`identified`イベントで返す
3. **卓の作成**: 新しく卓を立てる場合、識別済みのクライアントが`create_table`アクションを送信する(卓の設定を含む)。サーバーが「プレイルームID」を発行し`table_created`で返す。作成者はこのIDを参加者に共有する
4. **卓への参加**: 参加者は`join_table`アクションでプレイルームIDを指定して卓に加わる(再接続の場合はセッショントークンを添えて`join_table`し、そのまま復帰する)
5. サーバーがプレイヤーを卓に割り当て、`state_snapshot`イベントで現在の全体状況を返す
6. 参加者は`ready`でゲーム開始への参加を表明する(参加費チップ1枚はポット開始時に自動で徴収される)。卓の作成者は通常`ready`を送らず、参加者の準備が整った時点で`start_pot`を送ってポットを開始する(作成者自身は`start_pot`により自動的に参加者に含まれる)。作成者**以外**の全員が`ready`しても自動開始はせず、開始の判断は作成者に委ねられる。ただし作成者を**含む**全員が`ready`した場合は自動開始する(AIのみの卓が`start_pot`なしで始まる経路)。安全弁として、最初の`ready`から10分経過すると準備済みの参加者だけで自動開始する
7. ゲーム進行中、サーバーは状態変化のたびにイベントを配信し、必要なプレイヤーには`turn_prompt`等でアクションを要求する(タイムアウト秒数はサーバー設定値。人間/AIで別々に設定)
8. **切断**: 手番中に切断した場合、その手番はノーチェンジとして処理される(`docs/protocol/decisions.md`)。プレイヤーはゲームからは脱落しない
9. **再接続**: 同一セッショントークンで`join_table`すると、`state_snapshot`で最新状態を受け取り、そのままプレイに復帰する
10. **次ポット開始時の人数不足**: 必要なプレイヤーが揃わない場合、`game_ended`イベント(所持チップによる順位付き)を配信してゲーム終了
11. **ゲーム終了後の部屋の再利用**: 通常モードの卓は`game_ended`後に待機状態へ戻る(同じプレイルームIDのまま)。残ったプレイヤーが再度`ready`し、新規参加者も同じIDで`join_table`でき、作成者の`start_pot`で新しいゲーム(チップは初期値にリセット)を開始できる
12. **主催者の引き継ぎ**: `start_pot`の権限は「実効主催者」が持つ。卓の作成者が切断している間は、接続中の最も古くから参加しているプレイヤーに自動的に委譲され(`state_snapshot`の`creator_id`もその実効値を返す)、作成者が復帰すれば権限も戻る
13. 複数の卓は互いに独立して並行進行する。1つの卓の進行状況が他の卓に影響することはない

セキュリティ(不正なプレイルーム参加の防止、認証等)は本プロジェクトの規模・用途では考慮しない。プロトコルはAI実装側のバグ(不正な操作の送信など)に対処できる精度があれば十分とする。

## メッセージ形式(エンベロープ)

すべてのメッセージは以下の形式のJSONオブジェクト。

```json
{
  "type": "string (メッセージ種別)",
  "id": "string (メッセージ固有ID、任意)",
  "table_id": "string",
  "protocol_version": "string (例: \"1.0\")",
  "payload": { "...": "種別ごとのデータ" },
  "ts": "ISO8601 timestamp"
}
```

- クライアント→サーバーのアクションには `type` に加え、対応する `payload` を含める
- サーバー→クライアントのイベントも同形式。`payload`の内容は受信者によって内容が異なる場合がある(例: 自分の手札は本人にしか送らない)
- `protocol_version`は互換性チェック用。ゼミ生が個別に実装するAIクライアントがプロトコルの変更に追従できていない場合、サーバーはバージョン不一致を検出して`action_rejected`で明示的に伝える

## アクション一覧(クライアント→サーバー)

| type | 送信者 | 内容 | payload例 |
|---|---|---|---|
| `identify` | 全員 | 接続時の自己紹介。セッショントークンの発行を受ける | `{name, player_type}` |
| `create_table` | 卓を立てるプレイヤー | 新しい卓を作成し、プレイルームIDを発行してもらう。ゲーム開始時の設定を含む | 下記参照 |
| `join_table` | 全員 | プレイルームIDを指定して卓に参加(または再接続で復帰) | `{room_id, session_token?}` |
| `ready` | 参加者(human/ai) | ゲーム開始(最初のポット)への参加表明。ゲーム進行中の後続ポットは自動復活ルールで全員が自動参加するため再送不要(送っても無害・無視される)。`game_ended`後に同じ部屋で連戦する場合は改めて送る | `{}` |
| `dealer_ready` | 親のみ | 「どうぞ」宣言。配布後、自分の手札を確認したうえでディール開始を許可。親がクク札を持つ場合は、「どうぞ」の代わりに`cucco_declare`を送ってもよい(この待機中も親の宣言は即時有効) | `{}` |
| `cambio_declare` | 手番プレイヤー | 自分の右隣のプレイヤーとの交換を要求する(「カンビオ」) | `{}` |
| `no_change_declare` | 手番プレイヤー | 交換しない(「ノンカンビオ」) | `{}` |
| `cucco_declare` | クク札所持者 | ディール即終了。**ファイア・アンド・フォーゲット**: プロンプトへの応答ではなく、ディール中いつでも任意のタイミングで送信できる。サーバーは受信をフラグとして保留し、アトミックな処理の外側の直近の安全点で適用する(「クク宣言(非同期)」参照)。宣言を見送ることは単に送らないことで表現される(専用のパス操作はない) | `{}` |
| `continue_declare` | 子供の時間の敗者 | 規定チップを払って次ディールに参加継続するか、辞退するか | `{continue: boolean}` |
| `start_pot` | 卓の作成者のみ | 準備完了済みの参加者だけで今すぐポットを開始する(最小人数以上必要)。作成者自身は`ready`不要(`start_pot`の送信自体が参加表明として扱われ、自動的に参加者に含まれる) | `{}` |
| `result_ack` | 参加者(human) | `result_pause`(結果確認ウィンドウ)を確認したことの表明。着席中の接続済み人間プレイヤー全員が送ると、待機時間の残りをスキップして即座に進行する | `{}` |
| `effect_declare` | `effect_window`の宛先 | (宣言式ルールのみ)自分の特殊札の効果を宣言して交換を拒否する。宣言可能札を持っていない場合に送っても交換受諾として扱われる | `{}` |
| `effect_pass` | `effect_window`の宛先 | (宣言式ルールのみ)効果を宣言せず、交換に応じる。宣言可能札を持たない対象者はこれで交換を確定させる。タイムアウト・切断も同じ扱い | `{}` |

補足: 交換相手は常に「自分の右隣のプレイヤー」に固定されるため、`cambio_declare`に対象プレイヤーの指定は不要。馬・家の開示方法はゲーム開始時の設定(`create_table`のpayload)で決まるため、都度の選択アクションは存在しない。

`create_table`のpayload:

```json
{
  "mode": "normal | evaluation",
  "game_count": "number (evaluationモードのみ、自動連続実行するゲーム数)",
  "end_condition": "chips_zero | round_limit",
  "round_limit": "number (end_conditionがround_limitの場合)",
  "starting_chips": "number (デフォルト25)",
  "disqualified_card_disclosure": "immediate | deferred (下記3つの一括設定。デフォルトdeferred)",
  "joker_disclosure": "immediate | deferred (道化による失格者のカード開示タイミング。省略時はdisqualified_card_disclosureの値)",
  "human_disclosure": "immediate | deferred (人間による失格者のカード開示タイミング。省略時はdisqualified_card_disclosureの値)",
  "cat_disclosure": "immediate | deferred (猫による失格者のカード開示タイミング。省略時はdisqualified_card_disclosureの値)",
  "horse_house_reveal": "boolean (拒否時に馬か家かを開示するか)",
  "turn_timeout_human_sec": "number (デフォルト30)",
  "turn_timeout_ai_sec": "number (デフォルト10)",
  "cucco_window_timeout_human_sec": "number (デフォルト10)",
  "cucco_window_timeout_ai_sec": "number (デフォルト2)",
  "result_pause_sec": "number (デフォルト0、最大60。「オープン」直後(続行確認の前)とポット決着後に、結果を確認するための待機秒数。`result_pause`イベントとして全員に通知され、着席中の人間全員が`result_ack`を送れば残り時間をスキップする。評価モードでは無視される)",
  "effect_declaration": "auto | declared (デフォルトauto。declaredの場合、道化を除く特殊札(人間/馬/猫/家)の拒否効果は、交換を要求された保持者が`effect_window`で能動的に宣言した場合のみ発動する。宣言しなければ効果は発動せず交換が成立する。道化の受取失格と、山札から引かれた特殊札の効果は宣言不要で常に自動発動)",
  "discard_display": "grouped | pile (デフォルトgrouped。捨て札の表示スタイル。groupedは種類ごとの全一覧、pileは物理的な捨て山のように最後に捨てられた1枚のみを表示する。**表示上の卓設定であってルールではない** — 配信されるデータ自体は変わらず、`state_snapshot`の`discard_display`で全クライアントに共有される)",
  "ai_players": "[{policy, count}] (省略可。サーバー内蔵のAIプレイヤーをこの卓に着席させる。policyは方策名(always_change / always_no_change / matrix / counting_aggressive / counting_conservative)、countは人数。合計は最大参加人数-1まで — 作成者の席は必ず残る)"
}
```

`ai_players`で指定した内蔵AIは、**最初のセッションが`join_table`した時点で**着席する
(作成前に着席させると、内蔵AIの全員readyが作成者の参加前に自動開始条件を満たして
しまうため)。内蔵AIは外部AIクライアントと同一のプロトコル経路(identify →
join_table → ready → 各宣言)を通る通常の参加者であり、`state_snapshot`の
`seats[].player_type`は`"ai"`、表示名は`AI-<方策名>-<番号>`。ゲーム終了後も
退席せず、再び`ready`を送って連戦に応じる。作成者が観戦者(または評価モード)の
場合、参加資格者は内蔵AIのみになるため、内蔵AIの全員readyで自動開始する
(AI同士の対局を観戦する用途)。内蔵AIは切断しないため、**内蔵AIが2体以上残る卓は
人間全員が離脱しても自動終了しない**点に注意(卓の掃除は管理機能の対象、今後対応)。

`joker_disclosure` / `human_disclosure` / `cat_disclosure`は失格原因ごとに独立指定できるが、
`disqualified_card_disclosure`を指定すると未指定の個別フィールドのデフォルト値として使われる
(個別フィールドを明示した場合はそちらが優先される)。よくある「3つとも同じ設定でよい」場合は
`disqualified_card_disclosure`だけ送れば済み、原因ごとに変えたい場合だけ個別フィールドを使う。

## イベント一覧(サーバー→クライアント)

| type | 内容 |
|---|---|
| `identified` | `identify`への応答。セッショントークンを含む |
| `table_created` | `create_table`への応答。発行されたプレイルームIDを含む |
| `state_snapshot` | 現在の全体状況。内容は「公開ゲーム状態」を参照。参加時・再接続時・観戦開始時に送信 |
| `action_rejected` | AIプレイヤーが送った操作が現在の状態で不正だった場合の拒否通知(デバッグ用。人間向けUIでは基本的に発生しない) |
| `pot_started` | 新しいポット開始。参加者・各自の現在チップ数(絶対値)・最初の親(`dealer_seat`)・ポットの初期チップ数(`pot_chips`: 参加費+持ち越し分) |
| `deal_started` | 新しいディール開始。宛先プレイヤーには自分の手札を含める。残り山札枚数を含む |
| `dealer_ready` | 配布直後、親にのみ送られる「どうぞ」確認プロンプト(`timeout_sec`を含む)。親は`dealer_ready`アクション(またはクク保持なら`cucco_declare`)で応答する |
| `result_pause` | 結果確認ウィンドウの開始(「オープン」直後とポット決着後)。`timeout_sec`を含む。全員に配信され、着席中の人間全員の`result_ack`または時間切れで進行が再開する |
| `effect_window` | (宣言式ルールのみ)交換を要求された**全ての対象者**に送る個別通知(宣言可能札の保持者に限らない)。宣言可能札(人間/馬/猫/家)の保持者は効果を宣言するか選び、それ以外の札の保持者は「交換に応じる」を返すだけ。**対象を保持者に限定すると、応答までのラグの有無だけで特殊札の所持が卓全体に漏れる**ため、全員に同じ確認を挟んでタイミングを均一化する。`requester`と`timeout_sec`(`cucco_window_timeout_*`の値。名称は歴史的経緯)を含む。無応答は`effect_pass`扱い。宣言不能札の保持者からの`effect_declare`は交換受諾として扱う |
| `deck_reshuffled` | ポット途中で山札が尽き、捨て札から再構築されたことを通知。再構築後の残り枚数を含む |
| `turn_prompt` | 特定プレイヤーに手番が回ってきたことを通知(タイムアウト秒数を含む)。クク札所持者は、カンビオ/ノンカンビオの代わりに`cucco_declare`を送ってもよい(UI上は第3のボタン。非同期宣言なので厳密には「応答」ではなく、いつ送っても即時有効) |
| `no_change_declared` | あるプレイヤーがノンカンビオを宣言したことを全員に通知(カウンティング用の公開情報) |
| `turn_timeout_consumed` | タイムアウトによりノーチェンジとして処理されたことを通知 |
| `cucco_declared` | クク宣言が成立したことを全員に通知(誰が宣言したか) |
| `exchange_result` | 交換要求の結果(成立/拒否/連鎖/特殊効果の発動)。プレイヤー間交換では関係者本人にのみ交換されたカードの値を含める。**山札との交換(`deck_exchange_accepted`/`deck_draw_refused`)は引いたカード・手放したカードとも全員に公開**(物理卓では山札から引く行為も捨て札も見えているため) |
| `player_disqualified` | 途中失格イベント(誰が、どの効果で)。カードが即時公開の設定なら捨て札への追加(カード内容)も併せて通知 |
| `deal_opened` | 全員の手札を公開し、最弱者を判定した結果。公開された全員のカードを含む |
| `deal_result` | そのディールの敗者・支払ったチップ枚数・**各プレイヤーの現在チップ数(絶対値)**・敗者支払い後のポット枚数(`pot_chips`)・次の親・捨て札に加わったカード一覧 |
| `dealer_changed` | 親の交代を全員に通知(`player_id`=新しい親) |
| `continue_prompted` | 子供の時間の敗者に継続確認が行われることを**全員に**通知(`player_id`, `required_chips`)。当人に届く個別の`continue_prompt`と対になる公開情報で、他プレイヤーのUIが「◯◯の続行確認を待っています」を表示するために使う |
| `continue_prompt` | 子供の時間の敗者に、継続するか確認を要求(当人にのみ。`timeout_sec`を含む) |
| `chips_paid` | 敗者が支払いを済ませてポットに残ったことを全員に通知(`player_id`, `amount`, `chips_now`=絶対値) |
| `player_left_pot` | プレイヤーがそのポットから抜けたことを通知(復活辞退/支払い不能/大人の時間の脱落のいずれか、理由を含む) |
| `pot_won` | ポットの単独勝者が確定した瞬間の個別イベント(`winner`, `amount`, `chips_now`)。直後の`pot_result`集計と重複するが、ドメインイベントの逐次配信として先に届く |
| `pot_wiped_out` | 全滅によりポットが持ち越しになった瞬間の個別イベント(`amount`)。`pot_won`同様、直後の`pot_result`と対応する |
| `pot_result` | ポットの決着(勝者・総取り、または持ち越し)。**各プレイヤーの現在チップ数(絶対値)** |
| `game_ended` | ゲーム全体終了(チップ数による最終順位) |
| `evaluation_summary` | 評価モードで指定回数のゲームがすべて完了した際、集計結果を配信 |

## 公開ゲーム状態(Public Game State)

`state_snapshot`が保持・配信する状態の完全な内訳。各種イベントはこの状態への差分更新として扱われるが、**チップ数は常に絶対値で送る**(差分の積み上げによるクライアント側のズレを防ぐため)。

```json
{
  "table_id": "string",
  "mode": "normal | evaluation",
  "seats": [
    { "player_id": "string", "name": "string", "player_type": "human | ai", "chips": "number", "in_current_pot": "boolean", "connected": "boolean" }
  ],
  "spectators": ["player_id", "..."],
  "creator_id": "player_id (実効主催者。start_potを送信できるのはこのプレイヤーのみ。作成者の切断中は接続中の最古参プレイヤーに委譲される)",
  "ready_ids": ["player_id", "... (現時点でreadyを宣言済みの参加者一覧)"],
  "game_finished": "boolean (ゲームが終了済みか。game_ended配信後に再接続したクライアントへの通知手段)",
  "final_ranking": "[[player_id, chips], ...] | null (game_finishedがtrueの場合の最終順位)",
  "dealer_seat": "player_id",
  "current_turn_seat": "player_id | null",
  "pot_number": "number",
  "deal_number": "number",
  "pot_chips": "number (現在ポットに積まれているチップ枚数。物理卓では常に見えている公開情報)",
  "deck_remaining_count": "number",
  "discard_pile": [
    { "card": "string(ランク名)", "original_holder": "player_id | null", "discarded_via": "open | disqualification | deck_draw | dealer_swap", "discarded_at": "ISO8601" }
  ],
  "provenance_map": { "現在の所持者player_id": "その札の最初の持ち主player_id、山札から直接受け取った札の場合はnull" },
  "declarations_this_deal": [
    { "player_id": "string", "action": "cambio | no_change | cucco_declare", "via_timeout": "boolean", "ts": "ISO8601" }
  ],
  "your_hand": "string(ランク名) | null (この接続の本人にのみ含める。手札を持たない場合はnull)"
}
```

- `discard_pile`は捨て札に加わったカードの一覧。「猫」の効果に必要な`provenance_map`(元の持ち主の追跡)とあわせて全員に公開する
  - `discarded_via`が`deck_draw`の場合(山札から引かれたが親が受け取らなかったカード。`docs/rules/final_rules.md`「山札から引かれたが親が最終的に受け取らなかったカードの行き先」参照)、`original_holder`は`null`になる
  - `discarded_via`が`dealer_swap`の場合(親が山札交換に成功し、手放した元のカード。`docs/rules/final_rules.md`「親の山札交換が成立した場合の元のカードの行き先」参照)、`original_holder`は通常通りそのカードの最初の持ち主。なお、親が新たに受け取った山札由来のカードは、`provenance_map`上`null`(元の持ち主なし)として扱う
- `declarations_this_deal`は、そのディール中に行われた宣言(カンビオ/ノンカンビオ/クク宣言)の時系列一覧。カウンティングや行動履歴の把握に使う。`deal_started`で空配列にリセットされる。手番タイムアウトによる自動ノーチェンジも`action: "no_change"`、`via_timeout: true`として記録する(タイムアウトによる行動もカウンティング上見落とされないようにするため)
  - クク宣言を**見送ったこと**は記録されない(そもそも「見送る」という操作が存在しない — 宣言は保持者が任意のタイミングで`cucco_declare`を送るファイア・アンド・フォーゲット方式で、送らなければ何も起きない)。「誰がククを持っているか」が公開履歴から漏れることはない
- **自分の現在の手札**(`your_hand`)は、宛先が本人の場合にのみ`state_snapshot`に含める(他プレイヤー・観戦者向けには含めない)。切断中に交換要求のターゲットにされて手札が変わっていても、再接続時の`state_snapshot`でこのフィールドから現在の手札を復元できる。ディールに参加していない(まだ配られていない/そのディールから既に脱落した)場合は`null`
- 観戦者にも同じ公開ゲーム状態が送られる(`your_hand`を除く)
- **山札の再構築**: 山札が尽きて捨て札から再構築される際(ポット途中、`docs/rules/final_rules.md`8.)、`deck_reshuffled`イベントを配信する。このタイミングで`deck_remaining_count`は再構築後の枚数に、`discard_pile`は空になる(そのディールの最弱判定・途中失格による捨て札はこの再構築で山札に戻るため、以降のカウンティングはリセットされる)
- **ポット開始時のリセット**: `pot_started`で`deck_remaining_count`は44(フルデッキ)に、`discard_pile`・`declarations_this_deal`・`provenance_map`は空にリセットされる。`pot_started`には最初の親(`dealer_seat`、ランダムに決定)を含める

全滅による持ち越し後、所持チップ0枚のプレイヤーを除いた残りが1人になった場合は、ディールを行わず`pot_result`でその1人を勝者として即座に配信する。残りが0人になった場合は、全員のチップを含めて`pot_started`を再送し、「大人の時間」から次のポットを開始する(参加費は免除する。`docs/rules/final_rules.md`「チップ0枚の判定タイミング」参照)。

親がディール途中で失格した場合、山札交換役はその時点で手番順の最後尾になっている、かつ**まだ自分の通常の手番を消化していない**プレイヤーに引き継がれる。該当者がいない場合(親の失格が直前のプレイヤー自身の手番中に発生した場合など)は、山札交換を行わずそのまま「オープン」に進む(`docs/rules/final_rules.md`「親が途中失格した場合の山札交換役」参照)。プロトコル上は`turn_prompt`の宛先が動的に変わる、または`turn_prompt`自体がスキップされて`deal_opened`に進むだけで、新たなイベント種別は不要。

## クク宣言(非同期・ファイア・アンド・フォーゲット)

ルール上、クク宣言は自分の手番かどうかに関わらず、アトミックな処理中を除きディール中いつでも行える(`docs/rules/final_rules.md`)。プロトコルでは**非同期の宣言**として実現する。かつての方式(各処理の合間にサーバーが保持者へ`cucco_window`を送って応答を待つ)は廃止した — **保持者の応答を待つ間、卓全体の進行が停止するため、その待ち時間の発生自体から「クク保有者がいる」ことが全員に露見してしまう**ためである。

- クク保持者は、ディール中の任意のタイミングで`cucco_declare`を送信できる。サーバーはこれをプロンプトへの応答としてではなく**保留フラグ**として受け取り、直近の安全点(アトミックな処理の切れ目)で検証・適用する。**サーバーがクク保持者の判断を待つ局面は存在せず**、卓の進行テンポは保持者の有無と完全に無関係になる
- **他プレイヤーの手番の考慮時間中でも宣言は即座に成立する**(考慮時間はアトミックな処理ではない)。その手番は消滅し、ディールは直ちに終了する(「クク宣言は残りの手番進行を即座に停止」)
- **カード交換のアトミックな処理中(連鎖・効果解決を含む)に届いた宣言は、その処理の完了直後に適用される**(交換処理への割り込みはできない)。処理の結果ククが手元を離れていた場合、宣言は無効として黙って破棄される
- **親は「どうぞ」(`dealer_ready`)の場面で宣言できる**(親の通常手番は最後尾のため、他プレイヤーが動く前に宣言したい親の唯一の機会)。**親以外の宣言が効力を持つのは「どうぞ」以降**: 「どうぞ」前に届いた宣言は破棄されず保留され、「どうぞ」直後の最初の安全点で適用される
- **途中失格したプレイヤーはクク札を保持していても宣言権を失う**。該当する失格原因の開示設定が`deferred`でカードがまだ手元にある状態でも同様(`docs/rules/final_rules.md`)。失格者・非保持者からの宣言(保留フラグ)は検証時に黙って破棄される(`action_rejected`は返さない)
- 複数人がほぼ同時に宣言した場合、安全点での検証は決定的な席順(親から手番順)で行われ、最初の有効な1件が成立する(結果はどちらが処理されても同一。`docs/rules/final_rules.md`のクク宣言の効果を参照)
- クライアントUIは、クク札を保持している間**常設の「クク宣言」ボタン**を表示する(モーダルや割り込み通知は使わない)。自分の手札は本人にのみ配信されるため、ボタンの表示自体が他人に漏れることはない

## 情報可視性

| 情報 | 可視性 |
|---|---|
| 自分の手札 | 本人のみ |
| 他人の手札(ディール中) | 非公開 |
| 捨て札(「オープン」で公開されたカード) | 全員に公開 |
| カードの「元の持ち主」履歴(猫の効果用) | 全員に公開(常時参照可能) |
| 途中失格者のカード(道化/人間/猫による失格) | 最終的には必ず捨て札として公開されるが、公開タイミング(即時 or ディール終了時)は失格原因ごとに独立して設定による(`joker_disclosure` / `human_disclosure` / `cat_disclosure`) |
| 馬/家による拒否時、どちらの札かの開示 | ゲーム開始時の設定による(`horse_house_reveal`)。都度の選択はしない |
| 猫・人間による拒否 | 発動した時点で全員に判明(仕様上不可避) |
| 各プレイヤーのチップ枚数 | 全員に公開(常に絶対値で通知) |
| 親の位置、現在の手番 | 全員に公開 |
| 残り山札の枚数 | 全員に公開(`deck_remaining_count`) |
| そのディール中の宣言履歴(誰がいつカンビオ/ノンカンビオ/クク関連を宣言したか) | 全員に公開(`declarations_this_deal`) |

上記の公開情報はすべて「公開ゲーム状態」に定義され、`state_snapshot`および各種イベントを通じて全プレイヤー・観戦者に配信される。

観戦者は、対局プレイヤーと同じ「公開情報」のみを受け取る(誰の手札も見えない)。将来的に「特定プレイヤー視点での観戦(手札込み)」を追加する可能性はあるが、初期実装では非対応とする。

## タイムアウト・不正操作・切断

`docs/protocol/decisions.md`を参照。

- 手番(`cambio_declare`/`no_change_declare`)タイムアウトと切断はノーチェンジ扱い、再接続で復帰。再接続時、応答待ちのプロンプト(`turn_prompt`等)が残っていれば残り秒数付きで再送される
- 次ポット時の人数不足はゲーム終了+チップ順位判定。ただし通常モードでは、接続中のプレイヤーが2人未満になっても最大60秒間は再接続を待ってから終了判定する(リロードがポット境界をまたいだ場合の即死を防ぐ)
- 不正な操作: 人間向けUIはそもそも不正な操作を送信できない設計にする。AIには`action_rejected`で通知する
- 手番タイムアウト秒数はサーバー設定値。人間/AIで別々の値を持つ(`turn_timeout_human_sec` / `turn_timeout_ai_sec`)
- 手番以外のプロンプト(`dealer_ready`, `continue_prompt`, `effect_window`)にもタイムアウトのデフォルト動作を設ける。いずれもタイムアウトの計測は、対応する直前のイベントを受信した時点から始まる
  - `ready`はプロンプトではなく、個別のタイムアウトを持たない。ゲーム開始のゲートとしてのみ機能し(「接続・セッションライフサイクル」6.)、最初の`ready`から10分の安全弁で準備済みの参加者だけでの自動開始にフォールバックする。**ゲーム進行中の後続ポットへの参加に`ready`の再送は不要**(ドメイン層の自動復活ルールで全員が自動参加する。`docs/rules/final_rules.md`「次のポットへの参加」)
  - `dealer_ready`のタイムアウト: `deal_started`の受信時点から計測。タイムアウトすると自動的に「どうぞ」を宣言したものとして進める(親がクク札を持っていても、無応答ではクク宣言は行われない)
  - `continue_prompt`のタイムアウト: `continue_prompt`自体の受信時点から計測。タイムアウトすると辞退(そのポットから脱落)として扱う
  - `effect_window`のタイムアウト(宣言式ルールのみ): `effect_window`自体の受信時点から計測。`cucco_window_timeout_*`の値(名称は歴史的経緯)を使い、タイムアウトすると`effect_pass`扱い(効果は発動せず交換成立)
  - `cucco_declare`はプロンプトへの応答ではないため、タイムアウトの概念自体がない(送らなければ何も起きないだけ)

## AI専用高速評価モード

AIプレイヤーの性能評価・比較を目的とした、専用の卓モード。

- `create_table`時に`mode: "evaluation"`を指定して作成する。人間・観戦者は参加人数にカウントされず、観戦のみ可能(参加はAIのみ)
- 通常モードとの違い:
  - 人間の可読性のための待機・演出上のウェイトを省き、AIクライアントからの応答があり次第即座に処理を進める
  - タイムアウトは通常のAI用設定値をそのまま使う(評価目的だからといって不当に短くはしない。応答しないAIはそのバグとして扱う)
  - 1回の`ready`で複数ゲームを自動連続実行できる(`game_count`で指定)。各ゲームの終了(`game_ended`)後、チップを初期値にリセットして自動的に次のゲームを開始する
  - 座席順・各ゲームの最初の親をゲームごとにローテーション(またはランダム化)する。座席位置がCuccoの勝率に大きく影響するため、固定座席のまま連続実行すると評価結果に座席バイアスがかかる
  - 途中で`game_count`回に達したら卓を終了し、`evaluation_summary`イベントで全ゲームの集計結果(プレイヤーごとの勝率、平均順位、平均最終チップ数、失格率など)を配信する。座席ローテーションの内訳も含める
- 評価モードの卓でも観戦は可能(処理速度が速いため、閲覧用途としてはログを後から追う想定になる)

## 永続化・成績記録

- ゲーム終了(`game_ended`)のたびに、参加者の成績をサーバー側で永続的に記録する
- 記録内容: 卓ID、モード(normal/evaluation)、日時、参加者一覧(名前・種別)、最終順位、最終チップ数。評価モードの場合はゲームごとの記録に加えて集計結果も保存する
- 保存先はローカルファイル(SQLiteを想定)とし、iMac上に蓄積する。将来的にゼミ内で成績を参照・分析できるよう、簡単な参照手段(クエリ用スクリプト等)も用意する
- **行動ログとシャッフルシード**: 最終成績のみでは、対局のリプレイやルール上の疑義の検証、AIの戦略分析ができない。ゲームごとに、山札のシャッフルシードと、発生したイベント(アクション・結果)の時系列ログをJSON Lines形式で別途保存する。これにより後から対局を決定論的に再現できる
- この記録はクライアントへの応答とは独立したサーバー内部処理であり、新たなクライアント向けアクションは不要(必要であれば将来的に`get_results`のような参照APIを追加検討)

## 初期パラメータ(運用しながら調整可能)

- **手番タイムアウト**: 人間30秒 / AI10秒(サーバー設定値として変更可能にする)
- **プレイルームID**: 6桁の英数字コード(`0`/`O`、`1`/`I`など紛らわしい文字は除外)。卓作成のたびに新規発行し、使い回しはしない
- **卓の人数上限**: プレイヤー最大15人、観戦者最大30人程度
