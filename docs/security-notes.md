# セキュリティ運用メモ

本プロジェクトはゼミ内研究用のツールであり、要求仕様(`docs/requirements.md`
§3.4)では当初「学内LAN運用前提でセキュリティは考慮しない」としていた。しかし
Cloudflare Tunnel(trycloudflare.comの一時トンネル)でクライアント/サーバーを
**インターネットに公開して試験運用する構成**に移行したため、前提が変わっている。

この文書は、クライアント送信データ・無認証API経由での侵入・情報漏洩・チートに
関するリスクと、**コードで対処済みの範囲**および**運用側で対処が必要な残存リスク**
を整理する(サービス停止=DoS系は本プロジェクトの対象外。ただし後述のとおり
対策が「無い」ことは明記する)。

## コードで対処済み

以下は実装で恒久対応済み(該当テストあり)。

- **表示名のサーバー側検証**(`src/cucco/protocol/actions.py` `_require_name`):
  `identify`の`name`に長さ上限(24文字)・制御文字/書式文字(Unicode Cc/Cf)の
  拒否を追加。生WebSocketクライアントからの過長名・改行・RTLオーバーライド等に
  よる表示スプーフィングや、エスケープ漏れのある別実装クライアントへのXSS種の
  混入を、プロトコル境界で遮断する。
- **`create_table`数値設定の範囲チェック**(`src/cucco/domain/config.py`
  `GameConfig.__post_init__`):`starting_chips`≥1、`round_limit`≥1、各タイムアウトを
  `0 < t ≤ 3600秒`に制限。負値・0・巨大タイムアウトによる卓のクラッシュや
  「多時間デッドラインで同卓者を待たせ続ける」グリーフを防ぐ。
- **同名プレイヤーの参加拒否**(`src/cucco/server/dispatch.py` `_handle_join_table`):
  同一卓に既存プレイヤーと同じ表示名では参加できない。比較はNFKC正規化+
  casefold(`folded_name`)で行うため、全角/半角・大文字/小文字の差異
  (例:「Ａlice」「ALICE」「alice」)ですり抜けることもできない。
  **ベストエフォートの抑止**であり、後述のとおり多重着席の根本対策ではない
  (観戦者は席・手札を持たないため対象外)。

### 侵入面で確認済みの「安全側」性質(参考)

- サーバーが権威。不正な手はドメイン層の`IllegalAction`で拒否され、改造
  クライアントでも「非保持でクク宣言」等はできない。
- 手札の第三者漏洩なし。`your_hand`は本人宛のみ、交換結果の私的部分は当事者
  2名のみに配信。クク宣言を見送ったことは公開履歴に一切現れない(見送る
  操作自体が存在しない)。
- SQLインジェクションなし(`results_store.py`は全てパラメータバインド)。
  行動ログのパスに使う`room_id`はサーバー生成でパストラバーサル不可。`eval`/
  `exec`は不使用。

## 運用側で対処が必要な残存リスク

以下はコードだけでは塞げず、**運用ルール・構成・将来の認証導入**で対応する。

### 1. エンドポイントが無認証(インターネット露出)

サーバーは`0.0.0.0:8765`で待ち受け、接続時の認証は一切ない。トンネルURLを
知る者は誰でも接続・卓参加・観戦が可能で、守りは「ランダムなルームID
(32^6≈10億通り)」と「ランダムなトンネルドメイン名」という*秘匿性のみ*。
URLがSNS・スクリーンショット等で漏れると崩れる。

- **当面(quick tunnel)**: 共有URLをゼミ内クローズドに厳格運用。公開範囲を
  文書で明確化し、外部に貼らない。
- **恒久策**: 独自ドメイン取得後、named tunnel + **Cloudflare Access**で
  メール/Google認証を必須化(要求仕様§1.3「安定運用後に限定公開」に対応)。
  または`identify`/`join_table`に**共有入場シークレット**を1つ足すだけでも
  無差別接続はかなり潰せる。

### 2. 多重着席・共謀(チート)

一人が複数タブ/複数接続で別名参加すれば、その席数分の手札が見える。上記の
同名拒否は「同じ名前」しか止められず、別名を使う確信犯は防げない。場外ボイス
チャット等での共謀も同様にサーバー単独では防止不能。

- 本質的な防止には**参加者を個人と紐付ける認証**(上記1の恒久策)が必要。
- 当面は、卓作成者(=対局進行役)が参加者を目視で把握し、不審な多重参加が
  ないか運用で監視する。賭けチップの重みが大きい対局では特に留意。

### 3. セッションジャック(session_tokenは持参人型クレデンシャル)

再接続は`session_token`一致のみで席を引き継ぐ(値は128bit乱数で推測不能)。
つまりトークンは**それを持っている者が本人とみなされる持参人型クレデンシャル**
であり、**漏れれば即座に席の乗っ取り+その席の手札(`your_hand`)の閲覧**に
つながる。加えて実装上の注意点として:

- **再接続時に旧セッションを無効化しない**(`dispatch.py:_handle_join_table`は
  `existing.connection`を差し替えるのみ)。トークンを盗んだ接続と元の接続の
  両方が同じ`inbox`にアクションを送り込める状態になり、送信は新しい接続側に
  切り替わる。奪取者は手札を見つつ操作でき、被害者側の操作も混入して進行が
  乱れる。
- 現実的な漏洩面は、ゼミ共用PCの`localStorage`残存、画面共有/スクショでの
  開発者ツール露出、URL経由での不用意な受け渡し(本クライアントはURLに
  トークンを載せない設計だが、手動コピー運用で漏らさないこと)。

対策:

- 共用PCでは対局後にロビーの「忘れて新しく始める」でセッションを消す、または
  ブラウザの`localStorage`をクリアする運用を徹底する。
- 画面共有・スクリーンショット時に開発者ツールでトークンを露出させない。
- 恒久策としては、上記1の認証(Cloudflare Access等)を前段に置けば、トークン
  単独では成りすませなくなる。

### 4. 改造クライアント・バグクライアントの残存リスク

サーバーが権威であるため、**改造クライアントでもできないこと**がまず重要:

- 不正な手は打てない(ドメイン層`IllegalAction`+`_prompt`の型フィルタで拒否。
  「非保持でのクク宣言」「手番外の交換」等は通らない)。
- 他プレイヤーの伏せた手札は見られない(`your_hand`等の私的情報は宛先本人にのみ
  サーバー側で絞って配信。改造クライアントも受信バイト列は同じで、隠れた情報は
  一切増えない)。
- 他人になりすまして操作することはできない(相手の`session_token`が必要=上記3)。
- 他の卓に干渉できない(アクションはセッションの所属卓に限定。別卓のトークンを
  別ルームに提示しても拒否される。卓間は独立)。
- 不正JSON・バージョン不一致・未応答・切断は、それぞれ`action_rejected`・
  タイムアウトのノーチェンジ扱い・ノーチェンジ進行で吸収される(バグクライアント
  耐性は元の設計要件)。

一方で**残っているリスク**:

- **`player_type`が自己申告で未検証**(`identify`)。実害は2つ:
  (a) タイムアウト操作 — botが`"human"`を名乗れば手番30秒の長い持ち時間を
  得られる(本来AIは10秒)。(b) 評価モードへの偽AI混入 —
  評価卓(`mode: "evaluation"`)はAIのみが対局する前提だが、`"ai"`を名乗った
  人間クライアントが混じっても検出できず、ベンチマーク結果を汚染し得る。
- **DoS耐性なし**(下記6)。改造クライアントによる大量接続・大量送信・巨大
  ペイロード・卓の乱造は無制限。
- **ルームID列挙**(下記6と結合):レート制限が無いため、`join_table`を総当たり
  して稼働中の卓を探索し、観戦者/プレイヤーとして侵入する余地がある。ルームIDの
  秘匿性が唯一のアクセス制御であることに注意。

これらの根本対策も、突き詰めると認証(上記1)とレート制限(下記6)に帰着する。

### 5. 名前・卓の誤認(なりすまし)— 対処状況

「卓名の誤認」を誘う手法は**そもそも成立しない**。卓にユーザー設定の名前は無く、
識別子はサーバーが乱数生成する6桁のルームID(`registry.py`)だけで、攻撃者が
任意の卓名を付けて紛らわしく見せる余地がない(ただしルームID自体の秘匿性依存の
問題は上記1・下記6を参照)。

「ユーザー名の誤認」は**部分的に対処済み**:

- 対処済み — 完全一致の同名参加拒否、制御/書式文字(RTLオーバーライド・ゼロ幅等)
  の拒否、および全角/半角・大小文字の差異(NFKC+casefold)によるすり抜け防止。
- **残存** — クロススクリプトの同形異字(ラテン文字"a"とキリル文字"а"など、
  正規化しても別コードポイントのまま見た目が同一)は検出していない。別名を
  名乗る多重参加(上記2)と同様、表示名だけでの完全な同一性判定は原理的に
  困難であり、最終的には認証(上記1)と進行役の目視監視で補う。

### 6. DoS・レート制限が一切ない(対策なしを明記)

本プロジェクトはDoSを対象外としており、実際に**接続数・メッセージ流量・
ペイロードサイズ・卓の大量作成に対する制限は一切実装していない**。したがって:

- 単一クライアントからの接続フラッド/メッセージフラッド/巨大JSON送信、
  卓の無限生成などで、サーバープロセスを容易に飽和・停止させられる。
- 現状の防御はトンネル/ネットワーク層および「URLの秘匿」に依存しているだけで、
  アプリケーション層の防御はない。
- 公開範囲を限定し続けること、および異常時にサーバー/トンネルを即座に停止
  できる体制(手動運用)が、事実上唯一の対策である。恒久運用に移る場合は
  Cloudflare側のレート制限やアプリ層のスロットリング導入を別途検討する。

## まとめ

コード側は「侵入(注入・RCE)」「他人の手札の直接閲覧」「不正な手」「他卓への
干渉」については堅牢で、改造クライアントでもこれらは破れない。今回の対処で
クライアント送信データの検証漏れ(過長名・制御文字・全角半角/大小の同名すり抜け・
設定値の異常範囲)も塞いだ。

一方、コードだけでは塞げない残存リスクは以下で、いずれも**公開範囲の限定**と、
恒久運用時の**認証(Cloudflare Access等)**・**レート制限**の導入が前提となる:

1. 無認証エンドポイントのインターネット露出(守りはURL秘匿のみ)
2. 多重着席・共謀(別名を使う多重参加は防げない)
3. セッションジャック(session_tokenは持参人型、再接続で旧セッション無効化なし)
4. 改造/バグクライアントの残余(`player_type`未検証によるタイムアウト操作・
   評価モードへの偽AI混入)
5. 表示名のクロススクリプト同形異字によるなりすまし(卓名は存在しないため誤認対象外)
6. DoS・レート制限が皆無(ルームID列挙による卓探索を含む)

## 管理リスナー(v0.15.0)

サーバー管理面(卓一覧・進行状況・強制中止)は、公開ゲームポートとは**別の
WebSocketリスナー**として実装している(`cucco.server.admin`、デフォルト
`ws://127.0.0.1:8766`)。設計方針:

- **127.0.0.1バインド固定**。管理操作をトンネル(公開経路)に載せる案は不採用 —
  無認証のゲームポートと同じ経路に管理権限を並べると、トークン1本が唯一の
  防壁になってしまう(上記1.の問題を管理権限にまで拡大する)。管理は
  サーバーを動かしているマシン上のCLI(`python -m cucco.tools.admin`)から行う
- **管理トークン必須**。起動ごとに生成してログに出力(`--admin-token`で固定も
  可)。ローカルバインドが破られた場合(同一マシンの別ユーザー等)の第二防壁で
  あり、トークンがあるからといってこのポートを公開してよい理由にはならない
- **Cloudflare Tunnelの設定にこのポートを追加しないこと**。公開するのは
  ゲームポート(8765)のみ
- **管理コンソール(v0.17.0)**: 同じ管理リスナーがブラウザ用のHTMLコンソールも
  配信する(`http://127.0.0.1:8766/`)。ページのHTMLは`src/cucco/server/`に置く
  — `clients/`配下はGitHub Pagesへ自動公開されるため、管理UIを置いてはいけない。
  管理トークンはページに埋め込まず操作者が入力する(ポートに到達できただけでは
  操作できない状態を保つ)。WebSocketハンドシェイク以外のHTTPリクエストは
  コンソールのHTMLか404のみを返し、他のファイルは一切配信しない
