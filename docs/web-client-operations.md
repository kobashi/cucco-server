# ブラウザクライアント 操作手順書

人間向けブラウザクライアントを実際に動かして卓を立て、対局するための手順書。
クライアントは2種類あり、どちらも同じサーバー・同じ卓で相互に対局できる:

- **プレイ用クライアント(`clients/play/`)**: 卓を囲むレイアウトとカード・
  チップのアニメーションでゲームを楽しむための画面。通常の対局はこちらを推奨。
  右上のツールバーで以下を切り替えられる(設定はブラウザに保存される):
  - **メッセージ確認モード**(3段階): OFF(すべて自動で流れる)/ 最小
    (失格・クク宣言などディールを左右する重要な場面だけモーダルで確認。
    アニメと効果音で分かる交換・スキップ等は自動)/ フル(進行メッセージを
    1枚ずつモーダルで確認)。ボタンを押すたびに切り替わる
  - **子の時間 自動続行**: ONにすると、子供の時間(1〜3ディール目)の敗者に
    出る「続行しますか?」を、必ずチップを払って自動で続行する
  - **効果音** ON/OFF、**カード効果**一覧
- **リファレンスクライアント(`clients/web/`)**: プロトコル全機能の最小実装。
  動作検証・実装例向け(`clients/web/README.md`参照)

操作の流れ(卓作成・準備完了・主催者の開始・各種モーダル)は両者で共通。
実装が満たすべき要件は`docs/human-client-guide.md`を参照。

> [!NOTE]
> **サーバー側の運用**(起動オプション・ログ・安全な停止・管理CLI・GC・
> データ管理・バージョンアップ・トラブルシューティング)は
> [`docs/server-operations.md`](server-operations.md) が正。この文書では
> クライアント操作に必要な最小限だけ触れる。

## 1. 起動

1. サーバーを起動する(リポジトリルートで):
   ```bash
   source .venv/bin/activate
   python -m cucco.server.app
   ```
   `ws://localhost:8765`で待ち受ける。
2. 別ターミナルで、クライアント群を静的ファイルサーバーとして配信する
   (docrootは`clients/`。共有モジュール`clients/web-common/`を両クライアントが
   参照するため。ES Modulesは`file://`から直接読み込めない):
   ```bash
   python -m http.server 8000 --directory clients
   ```
3. ブラウザで `http://localhost:8000/` を開くとクライアント選択のランディング
   ページが表示される。リファレンスクライアントは`/web/`、プレイ用クライアント
   は`/play/`。デフォルトで`ws://<ページを開いたホスト名>:8765`に自動
   接続する。別ホスト/ポートのサーバーに繋ぐ場合は`?ws=host:port`付きURLを
   開くか、名前入力画面下部の「接続先を変更」から設定する。

### サーバーの停止

`SIGTERM`(通常の`kill`)または`SIGINT`(フォアグラウンド起動時はCtrl-C)で
安全に停止する。どちらもリスナーを閉じ、GCタスクを止め、接続中のクライアント
にはWebSocketのgoing-awayフレームが送られる(クライアントは「再接続中」表示に
なり、再起動後は`session_token`で復帰できる)。

```bash
pkill -f cucco.server.app          # ← -f 必須(既定でSIGTERM)
# または: kill $(pgrep -f cucco.server.app)
```

`pkill`は`-f`を付けないとプロセス名(`python…`)としか照合せず不発になる点に
注意。`kill -9`(SIGKILL)は不要(終了処理をスキップするだけ)。完了した対局は
その時点でSQLiteに保存済みなので、どの止め方でも記録は失われない(消えるのは
進行中の対局のみ)。

## 2. 卓を立てて参加者を募る

1. 名前を入力し、「プレイヤーとして参加」を選んで「つづける」
2. 「卓を作る」→設定フォーム(終了条件・開始チップ枚数・失格カードの開示
   タイミング・馬/家の開示有無・特殊札の効果の宣言式/自動・結果確認の
   待機時間)を入力して「作成する」。「宣言式」を選ぶと、道化を除く特殊札
   (人間/馬/猫/家)は保持者が交換要求時のモーダルで宣言した場合のみ効果が
   発動し、宣言しなければ交換がそのまま成立する。なお交換要求時のモーダルは
   **対象者の保持カードにかかわらず必ず表示され**、特殊札以外の保持者は
   「交換に応じる」を押して確定する(応答の間の長さから特殊札の有無を
   悟られないようにするため)
3. 発行されたプレイルームID(6桁)を参加者に共有する
4. 参加者は「プレイルームIDで参加する」からそのIDを入力し、「準備完了」を押す
5. 卓の作成者(主催者)は「準備完了」を押す必要はない。待機画面で参加者の
   準備状況(✅マーク)を確認し、揃った時点で「ゲームを開始する」ボタンを
   押すとポットが始まる(作成者自身は自動的に参加者に含まれる。作成者+
   準備完了1人の計2人から開始可能)

一人で対局したい場合は、卓の作成フォームの「AIプレイヤーを追加」で
サーバー内蔵AIの人数を指定するのが最も簡単(別プロセス不要)。指定した
AIは自分が卓に入った直後に着席し自動で「準備完了」になるので、あとは
「ゲームを開始する」を押すだけでよい。

外部プロセスのAIクライアントを使う場合は、卓を作った後、別ターミナルで
モックAIを参加させる(実装の中身は内蔵AIと同一):
```bash
source .venv/bin/activate
python -m clients.mock_ai.mock_ai --name Bot1 --room <プレイルームID> --policy matrix
python -m clients.mock_ai.mock_ai --name Bot2 --room <プレイルームID> --policy always_change
```
こちらも参加後すぐに自動で「準備完了」を送る。

複数人でブラウザから確認したい場合は、同じ手順で複数のタブ/ウィンドウを
開き、それぞれ別の名前で参加すればよい。

## 3. 対局中の基本操作

- **どうぞ**: 自分が親のとき、配布直後に表示される。押すと手番が始まる
  (クク札を持っていれば「クク宣言」ボタンも並ぶ)
- **カンビオ/ノンカンビオ**: 自分の手番のとき表示される。カンビオは右隣の
  プレイヤー(親の最終手番のみ山札)との交換を要求する(クク札を持っていれば
  「クク宣言」ボタンも並ぶ=ボタン3つ)
- **クク宣言**: クク札を持っている間、ディール中は**常設ボタン**として表示され
  続ける(自分の手番でも他プレイヤーの手番中でも同じボタン。自分の画面にしか
  出ない)。押すとディールが即終了する。宣言のために卓の進行が止まって待つ
  ことはない(進行の停止からクク保有者の存在が漏れない設計)
- **続行する/離脱する**: 子供の時間(1〜3ディール目)で敗者になったとき、
  必要チップ枚数と現在の残高が並んで表示される

いずれも自分が対象のときしかボタンが出ない(不正操作を未然に防ぐ設計)。

## 4. 確認しておきたいシナリオ

通常の対局フロー(配布→どうぞ→各自の手番→山札交換→オープン→結果)に
加えて、以下は個別に確認しておくとよい。

| シナリオ | 起こし方 | 確認するUI |
|---|---|---|
| 途中失格 | 人間・猫・道化による交換拒否が発生するまで対局する(AI混在で確率を上げるとよい) | 「このディールは失格しました。観戦のみです。」表示、以後そのディールで手番/クク宣言/どうぞの対象から外れること |
| クク宣言(手番) | クク札(44枚中2枚)を持つ手番(親なら「どうぞ」の場面)まで対局する | 「クク宣言」ボタンが手番/「どうぞ」の選択肢に並び、押すとディールが即終了すること |
| クク宣言(手番外) | クク札を持ったまま他プレイヤーの手番中に押す | 常設の「クク宣言」ボタンが手番外でも表示され続け、押すと(他プレイヤーの考慮中でも)ディールが即終了すること |
| 続行/離脱 | 子供の時間(1〜3ディール目)で敗者になる | 必要チップ枚数と現在の残高が正しく表示されること |
| 途中の再接続 | 対局中にブラウザタブをリロードする | 「前回の卓の続きがあります」→再接続後、現在の手札・進行状況が復元されること |
| 観戦 | 「観戦者として参加」で同じプレイルームIDから参加する | 手札欄・操作ボタンが一切出ないこと |
| ポット/ゲーム終了 | 誰かのチップが0枚になる、または既定ディール数に達するまで対局する | ポット結果・最終順位画面が表示されること |
| 連戦(部屋の再利用) | 最終順位画面で「この部屋で続けて遊ぶ」を選ぶ | 待機画面に戻り、再度準備完了→主催者の開始で新しいゲーム(チップリセット)が始まること。この間、同じプレイルームIDで新規参加者も加われること |
| 主催者の引き継ぎ | 主催者のタブを閉じたまま待機画面で待つ | 別のプレイヤー(最古参)の画面に「(主催)」表示と開始ボタンが現れ、そのプレイヤーが開始できること |

## 5. 学外公開しての試験運用(GitHub Pages + trycloudflare.com)

サーバーは変更不要(`src/cucco/server/app.py`はOriginチェックをしていないため、
別ドメインでホストしたクライアントからの接続をそのまま受け付ける)。**実際に
稼働させている構成**は以下(Cloudflare Pagesではなく最終的にGitHub Pagesを採用)。

- **クライアント群(`clients/`配下)**: GitHub Pagesで静的公開。
  `https://kobashi.github.io/cucco-server/`はクライアント選択のランディング
  ページで、リファレンスクライアントは`…/web/`、プレイ用クライアントは
  `…/play/`、共有モジュールは`…/web-common/`に配置される。
  `.github/workflows/deploy-pages.yml`(GitHub Actions、`clients/`配下の
  web関連の変更時に`main`へのpushで自動実行)が、Pythonパッケージを除く
  web部分だけを`_site/`に組み立ててPagesへデプロイする。リポジトリ設定は
  Settings → Pages → Source を「GitHub Actions」にしておく(一度だけ)
- **サーバー(`cucco.server.app`、8765番)**: ゼミのマシン上で起動したうえで、
  `cloudflared`の一時トンネル(trycloudflare.com、アカウント不要)で公開する:
  ```bash
  cloudflared tunnel --url http://localhost:8765
  ```
  実行するとランダムな`https://xxxx.trycloudflare.com`が発行される
  (`cloudflared`を再起動するたびにドメインは変わる)

クライアント(github.ioドメイン)とサーバー(trycloudflare.comドメイン)は
別ドメインになるため、クライアントに接続先を明示的に教える必要がある。
共有するURLに`?ws=`パラメータを付ける(1回開けば`localStorage`に保存され、
以後そのブラウザでは省略できる):

```
https://kobashi.github.io/cucco-server/?ws=xxxx.trycloudflare.com
```

(ランディングページが`?ws=`を各クライアントへのリンクに引き継ぐので、
このURL1つを共有すればどちらのクライアントでもそのまま接続できる。
特定クライアントへ直接誘導する場合は`…/web/?ws=…`・`…/play/?ws=…`)

`cloudflared`を再起動してドメインが変わった場合は、上記URLの`ws=`部分だけ
差し替えて共有し直せばよい。ブラウザから手動で接続先を変更したい場合は、
最初の名前入力画面下部の「接続先を変更(通常は不要)」からも設定できる
(`?ws=`を使えない・忘れた場合の予備手段)。

セキュリティ面(無認証公開・多重着席・トークン漏洩・DoS無防備など)は
[`docs/security-notes.md`](security-notes.md)にまとめてあるので、公開範囲を
広げる前に必ず目を通すこと。

## 6. セットアップ時に踏んだ落とし穴

このMac(ゼミ室サーバー)でGitHub Pages + cloudflaredを構築した際に実際に
つまずいた点。同じ構成を作り直す・別マシンに移す際の参考に残す。

- **`cloudflared`はHomebrewで入れる**(`brew install cloudflared`)。GitHub
  リリースから手動でバイナリを配置する方法もあるが、複数箇所(`~/bin`等)に
  重複させるとPATHのどちらが優先されるか分かりにくくなる。1系統に統一する
  こと
- **Homebrewのシェル初期化(`brew shellenv`)は`~/.zprofile`にしか入らないと、
  非ログインシェル(スクリプト実行・自動化ツール等)から`brew`/`cloudflared`/
  `gh`が見えない**。ログインシェルの対話ターミナルでは動くのに、自動化スクリプト
  からは`command not found`になる、という食い違いが起きたら、`~/.zshenv`にも
  同じ`eval "$(/opt/homebrew/bin/brew shellenv)"`を足すか、都度フルパス
  (`/opt/homebrew/bin/cloudflared`等)で呼び出す
- **`gh`のバイナリを入れ替える(手動配置→Homebrew)と、`~/.gitconfig`の
  `credential.helper`が古いパス(例:`~/bin/gh`)を指したままになり、
  `git push`が`No such file or directory`で失敗する**ことがある。この時、
  `~/.gitconfig`自体は編集せず、そのパスにHomebrew版へのシンボリックリンクを
  張って解決する方法もある(`ln -sf $(which gh の実体) ~/bin/gh`)
- **`.github/workflows/`配下のファイルをpushするには、`gh`が発行した資格情報に
  `workflow`スコープが必要**。無いと「refusing to allow an OAuth App to
  create or update workflow ... without `workflow` scope」でリモートに拒否
  される。`gh auth refresh -h github.com -s workflow`をブラウザ認可付きで
  実行して解決する(この操作は実行したマシンのトークンのみに影響し、他の
  マシンでの`gh`セッションには影響しない)
- **GitHub Actionsのワークフローファイル自体をpushしても、そのワークフローが
  `on.push.paths`で絞っている対象(例:`clients/web/**`)に変更が無ければ
  自動実行されない**。初回デプロイなど手動できっかけを作りたい場合は
  `gh workflow run <ファイル名>`で明示的にトリガーする
- **quick tunnel(`cloudflared tunnel --url`)とnamed tunnel
  (`cloudflared tunnel login`+`create`)は別物**。ログイン状態は前者の挙動
  (ランダムURL・アカウント非連携)に一切影響しない。Cloudflareダッシュボードでの
  接続状況モニタリングもnamed tunnel限定で、quick tunnelは表示されない
- ダッシュボードの「Install and run a connector」に出る接続コマンドは
  トークンを含む長い文字列(`eyJ...`)であり、トンネルの**UUID(ID)とは別物**。
  コピー先を間違えやすいので注意

## 7. 既知の制約

`clients/web/README.md`「既知の制約」を参照(成績・統計閲覧UIや評価モード
専用ダッシュボードは未実装、現在の手番表示はベストエフォート推測、
リロードをまたいだ場合の失格/未配布の区別が曖昧になる場合がある、など)。

成績の確認はブラウザではなく、サーバーを動かしているマシン上のCLIで行う:

```bash
python -m cucco.tools.stats                  # プレイヤー別の通算成績
python -m cucco.tools.stats --policies      # 内蔵AI方策別の成績
python -m cucco.tools.stats --player 名前    # 1人の直近ゲーム
python -m cucco.tools.stats --recent 5      # 直近5ゲームの結果
python -m cucco.tools.stats --evaluations   # 評価モードの実行一覧
```

読み取り専用なのでサーバー稼働中に実行してよい。出力にはプレイヤーの
表示名(ゼミ生の実名の場合がある)が含まれるため、**結果をファイルに
保存して公開リポジトリにコミットしないこと**。

## 8. サーバー管理(卓の確認・停止卓の中止)

> 詳細・全オプションは[`docs/server-operations.md`](server-operations.md) §6 を参照。

サーバー起動時に管理リスナーが`127.0.0.1:8766`で立ち上がり、管理トークンが
起動ログに出力される(`--admin-token`で固定、`--admin-port 0`で無効化)。
サーバーと同じマシン上で:

```bash
python -m cucco.tools.admin --token <トークン> list           # 全卓の一覧(無操作時間つき)
python -m cucco.tools.admin --token <トークン> status <卓ID>  # 卓の詳細(手札は含まれない)
python -m cucco.tools.admin --token <トークン> abort <卓ID>   # 進行中ゲームを強制終了して卓を閉じる
python -m cucco.tools.admin --token <トークン> remove <卓ID>  # ゲームの走っていない卓を削除
```

`abort`すると参加者には通常の`game_ended`(現在チップによる順位)が届き、
内蔵AIのタスクも停止する。管理ポートはローカル専用 — **Cloudflare Tunnelに
載せないこと**(`docs/security-notes.md`)。

**放置卓の自動掃除**: サーバーは定期スイープで放置卓を自動削除する
(v0.16.2〜)。対象は「実クライアント(人間・観戦者・外部AI)が誰も接続
していない状態が続いた卓」(**内蔵AIだけで連戦し続ける卓**もここに含まれる
— 観戦者がタブを閉じれば掃除される)と、クラッシュ/終了して放置された卓。
デフォルトは60秒間隔・無人10分/終了5分で削除。`--gc-interval 秒`で間隔変更、
`--gc-interval 0`で無効化できる。手動の`abort`/`remove`は即時掃除として
引き続き使える。
