# 人間向けUI実装ガイド

人間プレイヤー用のUI(Web/ネイティブ問わず、別プロジェクトとして実装予定。
`docs/requirements.md`「人間は別途用意するUIを通じて手動で操作する」)がこの
サーバーに接続するためのガイド。プロトコルの正式な定義は
`docs/protocol/design.md`、ゲームルールは`docs/rules/final_rules.md`を参照。
AIクライアント向けの`docs/ai-client-guide.md`と対になる文書で、プロトコルの
実体は同一だが、判断を下すのがロジックではなく人間である点、および表示・UX
上の要求が異なる点を中心にまとめる。

## 1. 接続からポット開始まで

手順自体はAIと同じ(`docs/ai-client-guide.md` §1参照)。ただし人間UI特有の
注意点が2つある。

1. `identify`は`player_type: "human"`を送る。観戦者として入るなら`"spectator"`
2. `session_token`は**ブラウザのタブを閉じても復元できる場所に保存する**
   (`localStorage`など、単なる変数保持では不十分)。人間はAIと異なりページ
   リロードや誤操作での切断が起こり得るため、再接続の実用性が重要になる

再接続時は、保存済み`session_token`を含めて`join_table`を送れば
`state_snapshot`(自分の現在の手札`your_hand`込み)が返り、そのまま復帰できる。
**サーバー側に「現在の状態を教えて」という専用リクエストは存在しない**ので、
UI側で状態がズレたと判断したら`join_table`の再送で復帰する設計にしておくこと。

## 2. UIが入力を求めるべき場面

AIの「判断ポイント」と同じ5箇所だが、人間UIでは**それぞれ明確な操作画面/UIパーツ**
として提示する必要がある。

| サーバーからの通知 | 状況 | 表示すべきUI | 送るアクション |
|---|---|---|---|
| `pot_result`(または卓参加直後の`state_snapshot`) | 次のポットが始まる前 | 「次のポットに参加する」ボタン(タイムアウトは人間30秒) | `ready` |
| `deal_started` | 自分が親、配布直後 | 「どうぞ」ボタン | `dealer_ready` |
| `turn_prompt` | 自分の手番 | 自分の手札を大きく表示し、「チェンジ」「ノーチェンジ」の2択 | `cambio_declare` / `no_change_declare` |
| `cucco_window` | クク札を保持している(かつ失格していない) | **急を要する通知**(下記) | `cucco_declare` / `cucco_pass` |
| `continue_prompt` | 子供の時間(1〜3ディール目)で敗者になった | 必要チップ数と現在の残高を並べて表示し、続行/離脱の2択 | `continue_declare`(`{continue: true/false}`) |

**`cucco_window`はタイムアウトが人間でも10秒しかない**(AIの2秒よりは長いが、
他の応答は30秒であることと比べると際立って短い)。物理卓での「今すぐ宣言する
か」の緊迫感を再現する意図的な設計であり、UI側もそれに応じた扱いが必要:

- 通常の手番待ちとは視覚的に区別する(モーダル、点滅、効果音など)
- 残り時間をカウントダウン表示する(人間はAIと違い応答時間を機械的に把握
  できないため、視覚的なタイマーが実質必須)
- **自分がクク札を保持していない限りこの通知は届かない**。届いた時点で
  「自分がククを持っている」ことが確定するので、UI上でそれ自体を強調してよい
  (他プレイヤーには一切知らされない情報)

上記以外の通知(`exchange_result`, `player_disqualified`, `deal_opened`,
`deal_result`, `pot_result`, `game_ended`など)は状態表示の更新のみで、
入力を求める必要はない。

## 3. 画面に常時表示すべき情報

`state_snapshot`の「公開ゲーム状態」(`docs/protocol/design.md`)を元に、
以下を常時見える位置に表示する。物理卓であれば当然目に入る情報なので、
どれか一つでも欠けると人間プレイヤーが物理卓より不利になる。

- **自分の現在の手札**(`your_hand`): 本人のみに配信される。観戦者には
  含まれないため、観戦画面では手札欄自体を出さない設計にする
- **全プレイヤーのチップ枚数**: 常に絶対値で届く。差分計算は不要、そのまま
  上書き表示してよい
- **親の位置・現在の手番**(`dealer_seat`, `current_turn_seat`): 座席表示に
  必須。自分の手番が来たことをUI上で強調する(音・振動・画面変化など)
- **残り山札の枚数**(`deck_remaining_count`): カウンティングの手がかりとして
  物理卓でも見える情報。`deck_reshuffled`でリセットされる
- **捨て札とその来歴**(`discard_pile`, `provenance_map`): 「猫」の効果
  (要求者が現在持っているカードの最初の持ち主が失格する)を予測するために
  人間プレイヤーも参照する情報。一覧表示だけでなく、「このカードは元々誰の
  ものだったか」を辿れるUIが望ましい
- **そのディール中の宣言履歴**(`declarations_this_deal`): 誰がいつカンビオ/
  ノンカンビオ/クク宣言をしたかの時系列ログ。**`cucco_pass`は含まれない**
  (誰がククを持っているか他プレイヤーに漏れないための意図的な仕様なので、
  UI側で「見送った人がいる」ことを推測できるような表示を作らないこと)

戦略の参考として`docs/rules/strategy_hints.md`と
`docs/rules/play_summary_granpere.md`(カードごとのチェンジ判断の目安)を
ゲーム内ヒント・チュートリアルとして人間向けUIに転用してもよい。

## 4. 典型的な1ディールの画面遷移(例)

4人卓、自分が2番手のプレイヤーである場合の例。`docs/ai-client-guide.md` §4の
メッセージフローと同一だが、UI視点での注釈を付す。

```
サーバー→全員: deal_started        (手札配布アニメーション、残り山札枚数更新)
サーバー→親:   cucco_window        (親のUIにのみ緊急通知)
親→サーバー:   cucco_pass          (通知を閉じる。他プレイヤーの画面には何も表示されない)
サーバー→(親以外のクク保持者): cucco_window (該当者のUIにのみ緊急通知)
親→サーバー:   dealer_ready        (「どうぞ」ボタン)
サーバー→全員: turn_prompt         (1番手のUIに手番強調表示、他は「待機中」表示)
1番手→サーバー: cambio_declare または no_change_declare
サーバー→全員: exchange_result     (交換演出、または「ノンカンビオ」ログ追加)
サーバー→(自分がクク保持なら): cucco_window
自分→サーバー: cucco_pass または cucco_declare
サーバー→自分: turn_prompt         (自分の手番強調表示)
自分→サーバー: cambio_declare または no_change_declare
サーバー→全員: exchange_result
... (残りのプレイヤーも同様に繰り返す) ...
サーバー→親:   turn_prompt         (親の最終手番、山札交換)
親→サーバー:   cambio_declare または no_change_declare
サーバー→全員: exchange_result     (山札からの交換。特殊札なら連鎖・失格演出)
サーバー→全員: deal_opened         (全員の手札を公開する演出)
サーバー→全員: deal_result         (敗者・支払いチップ・現在チップ数・次の親を表示)
(子供の時間の敗者がいれば) サーバー→敗者: continue_prompt
敗者→サーバー: continue_declare
```

途中で誰かが「人間」「猫」「道化」により失格した場合は`player_disqualified`が
割り込む。失格したプレイヤーのUIは以後の`turn_prompt`/`cucco_window`の対象
から外れるので、「このディールは観戦のみ」であることを明示する表示に切り替える
(`docs/rules/final_rules.md`「途中失格者はそのディールから完全に除外される」)。

## 5. エラー・異常系への対応

- **不正操作はUI側で未然に防ぐ設計にする**(`docs/protocol/design.md`
  「人間向けUIはそもそも不正な操作を送信できない設計にする」)。AI向けの
  `action_rejected`のような、プロトコルエラーを人間に生で見せる前提の設計には
  なっていない。具体的には、手番でない時は操作ボタン自体を無効化する、
  クク札を持っていないのに宣言できるUIを作らない、など。万一
  `action_rejected`を受け取った場合は、内部状態がサーバーとズレている
  バグの兆候として扱い、`join_table`(保存済み`session_token`付き)を再送
  して`state_snapshot`から状態を立て直す
- **タイムアウト**: 応答が間に合わなかった場合、`turn_prompt`はノーチェンジ
  扱い(`turn_timeout_consumed`)で自動進行する。UI側は「時間切れでノーチェンジ
  になりました」等のフィードバックを出し、プレイヤーが状況を把握できるように
  する。`cucco_window`のタイムアウトは無言でパス扱いになる(何も通知されず
  次に進む)ので、カウントダウン表示が特に重要になる(§2参照)
- **切断・再接続**: 手番中に切断してもゲームからは脱落しない(ノーチェンジ
  扱いで進む)。再接続後は`state_snapshot`の`your_hand`で現在の手札を確認
  してから操作可能にする。`your_hand`が`null`の場合は、まだそのディールで
  カードが配られていない(ディールとディールの間)か、そのディール中に既に
  失格して手札を失っている状態を意味するので、UIはこの2状態を区別して表示
  する(前者は「次のディールを待っています」、後者は「このディールは失格
  しました」)

## 6. 実装上の注意

- `protocol_version`をエンベロープに含めること。バージョン不一致は
  `action_rejected`になる
- 交換相手は常に「自分の右隣のプレイヤー」に固定されているため、UI上で
  交換相手を選択させる必要はない(`cambio_declare`にターゲット指定は不要)
- 馬・家の開示方法など、ゲーム開始時に決まる設定(`create_table`のpayload、
  `docs/protocol/design.md`参照)はディール中に変化しない。卓のルール説明
  画面などで一度提示しておけばよい
- **観戦モード**: 観戦者(`player_type: "spectator"`)は対局プレイヤーと同じ
  公開情報を受け取るが、誰の手札も見えない。観戦専用のUI(手札欄なし、
  入力ボタンなし)を用意すること。評価モード(`mode: "evaluation"`)の卓は
  人間・観戦者とも参加人数にカウントされず観戦のみ可能(`ready`を送っても
  参加者にならない)なので、AI同士の高速連続対局をログとして追いかける
  閲覧用途を想定した表示(進行が速いため、リアルタイム追従よりも直近の
  `deal_result`/`pot_result`の一覧表示が実用的)にするとよい
- 動作確認には`clients/stub/stub_client.py`(ターミナル対話クライアント)が
  参考になる。表示すべきイベントと入力タイミングの最小実装例として、
  UI実装前に一度動かしてメッセージフローを目で確認することを推奨する
